TLBEntry

Your TLBEntry class is a robust, canonical, and architecture-compliant Alpha AXP TLB/PTE entry for emulation or simulation. It reflects all the key design requirements for Alpha, VMS, and modern MMUs:

 

•All setters for addresses (virtual and physical) expect raw/unshifted values; shifting is only internal.

•Getters for VPN/PPN return the index (not a shifted address).

•All match/comparison functions are correct and parallel for VA/VPN and PA/PPN.

•Helper functions for page math (offset, mask, base, etc.) are complete and use correct shift.

•Utility and diagnostic helpers are well-labeled and separated.

•Static helpers for granularity, page size, and shift cover Alpha superpages.

•All protection/permission/flag checks are in place and easy to extend.

•Good use of inline documentation and comments referencing Alpha SRM and VMS Internals.

•Consistency between virtual and physical block APIs (so you can treat them identically in code).

•Key invariants and Q_ASSERTs on setters prevent hash corruption or accidental field mutation while hashed.The TLBEntry class ensures the following rule is applied uniformly and consistently when translating a Virtual Address (VA) to a Virtual Page Number and Physical Address (PA) to a Physical Page Number:

 

Architecture Goal Met:

“External code always passes the unshifted, raw address. TLBEntry always performs the shift. m_virtualPage and m_physicalPage are always stored as page numbers, never as addresses.”

 

 

Ensures:

•The CPU (and all software/hardware outside the TLB/PTE system) always works with the unshifted (raw) Virtual Address (VA) and Physical Address (PA).

•VA is used for all program-visible instructions, memory accesses, fetches, loads, stores, etc.

•PA is what actually goes to memory hardware or device.

•The TLBEntry class (whether used for TLB Instructions, Data, PageTable (PTE) or translation.  This TLBEntry class unit ensure a single bit-shift operation occurs.  

 

To Illustrate:

1.MMU/TLB translates VA → PA:

a.Internally, the TLB/PTE logic splits the VA into:

i.VPN (m_virtualPage): high bits (VA >> pageShift)

ii.Offset: low bits (VA & (pageSize-1))

b.Looks up VPN (+ASN) in TLB/page tables to find PPN.

c.Reconstructs PA:

PA = (PPN << pageShift) | (VA & (pageSize-1))

 

2.CPU sees only raw PA and raw VA.

a.CPU does not see VPN/PPN (page numbers) directly.

b.These are only used internally for lookups and translation.

c.If you “get” a VA from a TLBEntry, you use getVirtualAddressBase() (which re-shifts the VPN).

 

 

// Suppose m_pageShift = 13 (8KB pages)

quint64 va = 0xFFFF'FFFF'1234'5678;

entry.setVirtualAddress(va);                      // Internally: m_virtualPage = va >> 13

 

quint64 vpn = entry.virtualPageNumber();         // VPN (integer, e.g. 0x1FFFF_FFFF_FFFF)

quint64 vaBase = entry.virtualPageBaseAddress(); // VA base = VPN << 13

quint64 offset = entry.virtualPageOffset(va);    // Offset within page

bool match = entry.matches_VirtualAddress(va);   // True
