Interprocessor Interrupt (IPI) Stack Design

📄 Alpha Emulator — Interprocessor Interrupt (IPI) Stack Design

🎯 Purpose of the IPI Stack

 

The IPI (Interprocessor Interrupt) Stack is the mechanism by which one CPU in an SMP (Symmetric Multiprocessing) system can signal another CPU to perform an immediate or deferred action.

On Alpha hardware, this is accomplished via writing to the IPIR (Interprocessor Interrupt Register), which the target CPU monitors and responds to.

 

In the emulator, this mechanism is fully modeled and encapsulated by a stack of components that deliver, queue, and handle IPIs correctly.

 

🔷 Components of the IPI Stack

1️. Instruction Grain (PAL_MTPR_IPIR_InstructionGrain)

•Executes the PAL MTPR IPIR instruction.

•Decodes source CPU ID, target CPU ID, and IPI vector from the instruction.

•Constructs an IpiRequest and calls AlphaCPU::sendIpi() (or directly AlphaSMPManager::sendIpi()).

 

2️. AlphaCPU

•The initiating CPU receives the IpiRequest in its execution of the grain.

•Forwards the IpiRequest to AlphaSMPManager via sendIpi().

•For target CPUs, maintains a queue (m_pendingIpis) of incoming IPIs.

•Periodically (e.g., on instruction retire or via checkPendingIpis()) dequeues and processes pending IPIs by calling handleIpi().

 

3️. AlphaSMPManager

•Acts as the dispatcher and coordinator for SMP operations.

•Validates the source and target CPU IDs.

•Logs and enqueues the IpiRequest in its internal queue.

•Immediately delivers the IPI to the target CPU’s receiveIpi() method.

•Optionally emits Qt signals or updates statistics counters for monitoring and diagnostics.

 

4️. Target AlphaCPU

•receiveIpi() enqueues the incoming IpiRequest.

•handleIpi() interprets the IpiVector and performs the appropriate action:

oTLB_SHOOTDOWN → invalidates TLB.

oRESCHEDULE → triggers scheduling change.

oHALT → stops execution.

oDEBUG_BREAK → enters debug state.

•Logs and optionally sends an acknowledgment to the originating CPU or manager.

 

🔷 Data Types

IpiVector

Strongly-typed enum class representing the type of IPI:

 

enum class IpiVector : quint8 {

    TLB_SHOOTDOWN,

    RESCHEDULE,

    HALT,

    DEBUG_BREAK,

    CUSTOM1,

    INVALID = 0xFF

};

 

IpiRequest

Encapsulates the IPI information:

 

 

struct IpiRequest {

    quint16 sourceCpuId;

    quint16 targetCpuId;

    IpiVector vector;

    quint64 timestamp; // Optional for profiling

};

 

🔷 Flow of an IPI

1️. Source CPU executes the PAL_MTPR_IPIR instruction.

2️. An IpiRequest is constructed and sent to the SMP Manager.

3️. SMP Manager delivers the IPI to the target CPU.

4️. Target CPU enqueues the IPI.

5️. Target CPU eventually calls handleIpi() to act on the IPI.

 

🔷 Optional: Feedback Loop

If required, the target CPU can notify the SMP Manager or the source CPU that the IPI has been handled, for example by sending a CUSTOM_ACK vector or setting a flag. This is optional and OS/PAL dependent.

 

📋 Benefits of This Design

✅ Strongly-typed vectors avoid magic numbers.

✅ Clearly-separated responsibilities (grain → CPU → manager → CPU).

✅ Safe, thread-aware queueing with QMutex protection.

✅ Logs and statistics support debugging and profiling.

✅ Flexible for both standard and custom IPIs.
