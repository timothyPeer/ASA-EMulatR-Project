Overview

This section describes the unified exception handling model in the ASA-EMulatR. All exceptions—arithmetic, floating-point, privilege, memory, and system—are detected at the instruction grain level and processed in a way that mirrors Alpha AXP architecture. Each exception, including integer overflow, division by zero, reserved operand, and standard OS/PALcode patterns, is reported to the AlphaCPU via canonical exception vectors and typed records.

 

Exception detection and propagation

 

•The grain detects faults (overflow, divide by zero, reserved operand, unaligned access, underflow, inexact) during instruction execution.

•The exception status is set and cleared on the AlphaProcessorContext via the OptimizedExceptionStatusRegister, following Alpha AXP standards.

•Whether an exception is propagated as a trap is determined by both the FPCR trap enables and the OptimizedExceptionStatusRegister at the moment of exception, ensuring strict architectural compliance.

 

Lifecycle of an Exception (Alpha-Style)

Stage-by-stage, the exception handling proceeds as follows:

 

Exception occurs:

•Responsible entity: Instruction grain (e.g., ADDG_InstructionGrain).

•Effect: Bit set in status register (OptimizedExceptionStatusRegister).

 

Trap raised:

 

•Responsible entity: AlphaCPU (raises, vectors to PALcode).

•Effect: Bit remains set until handled.

 

Handler inspects:

•Responsible entity: PALcode or OS kernel handler.

•Effect: Status bit remains set for inspection/logging.

 

Handler clears:

•Responsible entity: OS/PALcode (clears by explicit write).

•Effect: Bit cleared in status register.

 

Resume execution:

•Responsible entity: AlphaCPU/handler.

•Effect: Status reflects the current exception state.

Canonical Exception Flow:

 

•The grain executes and detects a fault (e.g., integer overflow, memory fault).

•The grain notifies AlphaProcessorContext, e.g., by calling AlphaCPU->RaiseException() or setting bits in AlphaProcessorContext->OptimizedExceptionStatusRegister().

•AlphaProcessorContext  updates architectural registers:

•Sets bits in ESR (OptimizedExceptionStatusRegister)

•Updates relevant IPRs (e.g., EXC_SUM, EXC_ADDR, etc.)

•AlphaCPU saves processor state, pushes a trap frame, vectors to the handler (PALcode/OS).

Synchronous (Immediate) Exceptions:

The emulator immediately raises exceptions for:

 

•Privilege violations (PRIVILEGE_VIOLATION)

•Reserved operand faults (RESERVED_OPERAND_FAULT)

•Invalid instruction (OPCDEC_FAULT)

•Alignment faults (UNALIGNED_ACCESS)

•Floating-point enable faults (FEN_FAULT)

•Integer overflow, division by zero

•Any error where the trap condition is enabled

 

All synchronous exceptions are passed to AlphaProcessorContext via AlphaCPU::RaiseException.

 

Deferred (“Sticky”) Exceptions (Floating-Point):

•For floating-point instructions, the trap enable in FPCR is checked (e.g., TRAP_INEXACT_ENABLE).

•If enabled, exception is raised immediately.

•If not enabled, the “sticky” bit (e.g., STATUS_INEXACT_OCCURRED) is set in FPCR for later inspection.

 

Floating-Point Exception Handling:

•The grain calls a handler (e.g., checkFloatingPointExceptions) to check NaN/invalid, sqrt of negative, division by zero, etc.

•Handler raises exception or sets sticky as required.

 

Exception Classes and Typing:

•All exceptions inherit from QException (base: ASAException), and carry an ExceptionInfo record with category, detail, and diagnostic message.

•All exceptions are pushed to a thread-safe StackManager (as ExceptionFrame records), enabling unwind and resume logic per Alpha AXP requirements.
