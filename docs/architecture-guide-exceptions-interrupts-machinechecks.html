Architecture-Guide

1. Exception Types & Classification

 

•Synchronous (Exceptions): Caused directly by the current instruction (e.g., arithmetic overflow, access violation).

•Asynchronous (Interrupts): External or deferred (e.g., device interrupt, clock).

•Machine Checks: Hardware or fatal system errors (ECC, bus faults).

•Each has unique vectoring and handling paths.

 

2. Where Exceptions Are Hosted / State Ownership

 

•AlphaCPU

oOwns and updates primary exception state:

•Exception vector registers

oProgram counter (PC), status register (PS), IPL

•FPCR (for floating point traps)

oResponsible for raising, saving context, and vectoring to PAL handler

•AlphaProcessorContext

oHolds architectural register state at the moment of exception

oCan provide save/restore, but does not decide when to handle (AlphaCPU does)

•InstructionGrain

oDetects and signals exceptions on per-instruction basis, but does not process or resolve

 

Exception Processing Pipeline: Phases & Activities

A. Cause

 

•Exception source can be:

oCurrent instruction triggers (e.g., invalid opcode, overflow, unaligned access)

oMemory system faults (TLB miss, access violation, page fault)

oAsynchronous (interrupt controller posts a request)

oHardware detection (machine check)

 

•Detection occurs at:

oInstructionGrain (AlphaCPU::CheckForExceptions, validate operands, check for overflow, etc.)

oAlphaMemorySystem (page permission, TLB/translation faults)

oAlphaCPU (privilege, current mode, pending interrupts, fatal errors)

oDevice (for external interrupts)

 

B. Effect

 

•Upon detection:

oException state recorded (faulting PC, instruction, status, address, summary code)

oCurrent PS/PC and possibly general registers are saved to stack/context frame

oProcessor switches to Kernel mode and raises IPL as needed

oVectors to the appropriate PALcode entry point for the exception type

 

•For interrupts:

oRaises IPL, disables lower/equal priority events, vectors to PAL interrupt handler

 

C. Handling

 

•PALcode takes control at vector entry

oSaves additional processor context (if needed)

oDecodes exception cause and dispatches to OS-level handler

oHandles (or delegates) specific actions (e.g., fix TLB, signal process, emulate unaligned access)

oMay decide to resume, signal, or kill process depending on exception class

•AlphaCPU provides entry context, resumes on REI instruction

•Machine Checks: handled with highest IPL; correctable vs. fatal paths

 

D. Return to Processing

 

•Return From Exception is performed by the REI instruction:

oRestores PC, PS, IPL, registers from saved context

oReturns to previous mode (user/kernel) as appropriate

oIf exception was not recoverable, OS or PALcode may kill process or halt system

 

4. Exception Vectoring and Dispatch

 

•Vectors are assigned per-exception type (see dispatch tables in PALcode, e.g., for access violation, TLB miss, unaligned, arithmetic, machine check, etc.).

•Vectoring is handled by AlphaCPU, with all further processing in PALcode (OS layer).

•For interrupts, highest-priority pending interrupt is delivered when allowed by IPL.

 

5. Exception Context and Status Tracking

 

•AlphaCPU maintains:

oFault address, summary/status, saved PC/PS, IPL, FPCR

oSpecial fields for machine check (syndrome, error code)

•AlphaProcessorContext may provide register snapshot for restoration

•FPCR: Floating point status and trap flags may be updated or cleared by exception handler or REI

•Status is critical for precise vs. imprecise exception reporting (e.g., FP traps may be imprecise unless forced by TRAPB)

 

6. Key Considerations

 

a. Imprecise Exceptions

•Some traps (especially arithmetic) may be delayed/pipelined for performance reasons; OS may insert TRAPB to force precision.

 

b. Nested Exceptions & IPL

 

•While handling one exception, new exceptions (e.g., higher-priority interrupts) may be allowed, depending on raised IPL.

•IPL managed by AlphaCPU and interrupt controller; restored by REI.

 

c. Fault Recovery vs. Process Termination

 

•Handlers may fix faults (e.g., page in a memory page, handle copy-on-write), then resume.

•Some faults (reserved operand, unhandled machine check) may require process termination or system halt.

 

7. Roles by Component

 

•AlphaCPU

oDetects, records, vectors, and initiates exception handling

oManages processor status (PS, IPL, PC)

oInitiates context save/restore via PAL

 

•AlphaProcessorContext

•Provides register state and restore-on-return capability

 

•InstructionGrain

oDetects instruction-local exceptions and reports them to AlphaCPU

 

•AlphaMemorySystem

oDetects and reports address translation and protection exceptions

 

•InterruptController

oManages asynchronous event queue, arbitrates IPL, delivers interrupts to AlphaCPU

 

•PALcode

oReceives control at exception vector, saves full or minimal context, handles or delegates, signals user/OS, returns via REI_InstructionGrain

 

8. Typical Exception Handling Sequence

 

•Fault/interrupt detected by grain/memory/CPU/hardware

•AlphaCPU records exception details, sets PS/PC/IPL, and vectors to PALcode entry

•PALcode saves context, determines type/cause, handles (emulate, signal, recover)

•OS may be involved for page-in, signal delivery, process scheduling, etc.

•PALcode restores state, returns via REI_InstructionGrain, CPU resumes execution (or signals process/halts)

 

9. Return to Processing / REI

 

•REI instruction restores saved context (PS, PC, mode, IPL)

•Resumes execution at the point of interruption or next valid instruction

•May resume in user, kernel, or PAL mode depending on the type of exception and handler action

 

Implementation Notes

 

•Exception cause/effect tracking must be precise for some classes (memory, unaligned access); may be imprecise for FP/arith.

•All privilege transitions (user->kernel, kernel->PAL) and IPL changes are managed through exception/interrupt mechanisms.

•Test and validation (Chapter 14) requires explicit paths for each exception type, thorough vector/dispatch mapping, and state restoration checks.
