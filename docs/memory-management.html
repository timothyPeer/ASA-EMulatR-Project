Memory Management

 

===============================================================================

ASA AXP Emulator — Memory Management System Reference

===============================================================================

 

Overview

--------

This document describes the memory management system in the ASA-EMulatR, 

with emphasis on the AlphaMemorySystem class, its relationship to AlphaCPU, 

the MMU, caches, TLBs, and MMIO interfaces leading to the device busses.

 

The emulator models the memory hierarchy and device access pathways of 

an Alpha AXP system as described in the Alpha AXP System Reference Manual.

 

===============================================================================

Architecture Layers

-------------------

 

The memory and device access path is divided into these layers:

 

1.1CPU Core (AlphaCPU)

    → Generates virtual addresses, instruction fetches, data loads/stores.

2. Memory Management Unit (MMU)

    → Translates virtual → physical addresses.

    → Enforces page protections and privilege levels.

    → Integrates Translation Lookaside Buffers (TLBs).

 

3. Unified Caches (UnifiedDataCache, UnifiedInstructionCache)

    → Cache physical addresses and reduce memory latency.

    → Support LRU, LFU, or other replacement policies.

    → Enforce cache coherency across CPUs (SMP).

 

4. AlphaMemorySystem

    → Centralized owner of memory.

    → Handles physical RAM, page tables, MMIO regions.

    → Abstracts both normal memory and MMIO into a coherent interface.

 

5. MMIO (Memory-Mapped I/O)

    → Maps device registers into address space.

    → Routes accesses to device emulation classes (e.g., NIC, SCSI, UART).

 

6. Device Busses

    → Emulate PCI, ISA, or custom busses.

    → Manage device discovery, IRQ routing, and DMA.

 

===============================================================================

AlphaMemorySystem

-----------------

The `AlphaMemorySystem` sits at the heart of the emulator’s memory subsystem.  

It implements the physical memory abstraction, manages page mappings, 

and serves as the interface between the CPU and all memory-resident or MMIO resources.

 

### Responsibilities:

✔ Own and manage host-allocated RAM (system memory).  

✔ Provide methods to read/write memory at virtual and physical addresses.  

✔ Integrate with the MMU for page table management.  

✔ Detect and trap invalid memory accesses.  

✔ Manage MMIO regions and route to appropriate devices.  

✔ Support SMP coherency hooks where needed.

 

===============================================================================

Interaction: AlphaCPU → Memory

-------------------------------

 

At runtime, the AlphaCPU executes instructions and performs memory operations:

 

1️. Instruction Fetch

    - PC → MMU → Instruction Cache → AlphaMemorySystem → RAM/MMIO

    - Returns 32-bit or 64-bit instruction word to CPU.

 

2️. Data Load/Store

    - Virtual Address → MMU (translation) → Data Cache → AlphaMemorySystem.

    - Reads or writes data, respecting privilege and alignment.

 

3️.MMIO Access

    - Virtual Address (in MMIO range) → AlphaMemorySystem detects MMIO.

    - Routed to the appropriate device handler.

 

===============================================================================

AlphaMemorySystem Interface

---------------------------

 

Key methods (example signatures):

 

    bool writeVirtualMemory(cpuId, vAddr, data, size, pc);

    bool readVirtualMemory(cpuId, vAddr, &data, size, pc);

 

    bool writePhysicalMemory(pAddr, data, size);

    bool readPhysicalMemory(pAddr, &data, size);

 

    void mapRegion(base, size, type);  // Maps RAM or MMIO

    void unmapRegion(base, size);

 

    void registerMMIODevice(MMIODevice* device, baseAddr, size);

 

===============================================================================

MMU & TLB Integration

---------------------

The AlphaMemorySystem works in conjunction with the MMU and TLB system to 

translate virtual addresses to physical addresses:

 

- TLBSystem maintains per-CPU TLBs for fast address translation.

- On TLB miss, page tables (managed by AlphaMemorySystem) are consulted.

- Faults (page fault, protection fault) are reported back to CPU.

 

===============================================================================

Unified Caches

--------------

The UnifiedDataCache and UnifiedInstructionCache sit between the CPU 

and AlphaMemorySystem, caching physical memory accesses.

 

- Support L1/L2/L3 with configurable sizes, associativity, and replacement.

- Backed by AlphaMemorySystem.

- Support SMP coherency protocols (MESI, MOESI, etc.).

 

===============================================================================

MMIO & Device Busses

--------------------

The upper region of the memory space is reserved for MMIO.

 

- AlphaMemorySystem maps these regions to MMIODevice handlers.

- Device handlers expose memory-mapped registers and respond to reads/writes.

- MMIO accesses are routed through MMIODeviceManager → PCI/ISA busses → Devices.

 

Example flow:

    CPU → MMU → MMIO range → AlphaMemorySystem → MMIODevice → Device Emulation

 

===============================================================================

Summary of Components

---------------------

 

| Component               | Role |

|-------------------------|-----------------------------------------------|

| AlphaCPU                | Issues memory operations |

| MMU & TLBSystem         | Translate virtual → physical addresses |

| UnifiedDataCache        | Cache for data memory |

| UnifiedInstructionCache | Cache for instruction memory |

| AlphaMemorySystem       | Owns physical memory, MMIO, page tables |

| MMIODeviceManager       | Routes MMIO accesses to devices |

| PCI/ISA busses          | Host attached devices |

 

===============================================================================

Benefits of This Model

-----------------------

✓ Accurate emulation of Alpha AXP memory hierarchy.  

✓ Support for SMP & cache coherency.  

✓ Unified view of RAM & MMIO.  

✓ Modular & extensible for additional devices & busses.  

✓ Compatible with exception & trap handling when faults occur.
