Grain Execution - Store Instruction

This section documents the control and data flow for store instruction grains (such as STT_InstructionGrain::execute()) in the Alpha AXP emulator.

Store grains are responsible for decoding operands, calculating effective virtual addresses, reading source values from registers, and writing data to guest memory through the full memory management and translation hierarchy.

 

When a store instruction executes:

•The grain computes the effective virtual address (base register + signed displacement).

•Reads the value to be stored from the appropriate register (FP or integer).

•Calls into the virtual memory system to perform the store operation.

 

The memory system:

•Translates the virtual address to a physical address using the TLB ( TLBEntry) and, if needed, the page table.

•Handles TLB misses, page faults, and permission checks in compliance with Alpha hardware behavior.

•Performs the store to physical memory, supporting split operations for multi-word (e.g., TFloat/128-bit [dt_tFloat] ) values.

 

This process ensures architectural accuracy for address translation, TLB management, fault handling, and logging. It also provides a faithful emulation of how stores, faults, and multi-word operations work on real Alpha hardware and operating systems. The subsequent sections detail each step, from operand decode through TLB resolution, to final store in memory and post-execution logging.

 

Instruction Grain (e.g., STT_InstructionGrain::execute())

•The instruction implementation begins by:

oReading the base register (Ra)

oComputing the effective virtual address (VA) using base + signed displacement.

oReading the value to store (from FP or integer register, depending on instruction)

•Calling AlphaMemorySystem::writeVirtualMemory()

 

quint64 baseAddr = context->readIntReg(m_ra);

quint64 virtualAddr = baseAddr + static_cast<qint64>(m_signedDisplacement);

 

dt_tfloat tval = context->readFpRegRawTFloat(m_rc);

memorySystem->writeVirtualMemory(cpuId, virtualAddr, tval.low64(), 8);

memorySystem->writeVirtualMemory(cpuId, virtualAddr + 8, tval.high64(), 8);

 

AlphaMemorySystem::writeVirtualMemory()

•This function is the mirror of readVirtualMemory(), and:

oPerforms TLB lookup for VA → PA translation.

oIf not found, triggers TLB miss handler.

•On success, invokes writePhysicalMemory() with physical address.

 

TranslationResult tr = tlbSystem->translateAddress(va);

if (!tr.isValid()) {

    handleTLBMiss();  // May call page table walker

    return false;

}

writePhysicalMemory(tr.physicalAddress(), data, size);

 

TLBSystem::translateAddress()

•Returns cached translation (VA → PA) if TLB hit.

•Otherwise, calls back into memory system to handle the miss.

 

TLBEntry* entry = findDataTLBEntry(cpuId, va, asn);

if (!entry) return TranslationResult(/* invalid */);

return TranslationResult(entry->getPhysicalAddress(va), ..., true);

 

AlphaMemorySystem::handleTLBMiss()

•Walks the page table to resolve the VPN to PPN.

•Typically done via SimplePageTable::lookup().

•If found, installs the result into the TLB.

 

quint64 vpn = va >> 13;

PTE entry = simplePageTable->lookup(vpn);

quint64 pa = (entry.ppn << 13) | (va & 0x1FFF);

installTLBEntry(cpuId, va, pa);

 

// TODO DEPRECATED SimplePageTable::lookup()

•Key function for VA→PA resolution:

oExtracts the VPN: vpn = va >> 13

oLooks up vpn in QHash<vpn, PTE>

oPTE gives physical page number (PPN)

•Returns translated PA offset: ppn << 13 | (va & 0x1FFF)

 

Final Execution Flow Summary (for STT, STQ, etc.)

InstructionGrain::execute()

     ↓

AlphaMemorySystem::writeVirtualMemory()

     ↓

TLBSystem::translateAddress()

     ↓ (miss?)

AlphaMemorySystem::handleTLBMiss()

     ↓

SimplePageTable::lookup()

     ↓

(returns PA, updates TLB)

     ↓

AlphaMemorySystem::writePhysicalMemory()

     ↓

Store completes to memory

 

Additional Architectural Notes

Fault Detection:

•Page faults or protection violations can be triggered in writeVirtualMemory or during PTE lookup if access is not allowed.
