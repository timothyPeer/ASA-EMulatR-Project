FPArithmetic Module Overview

FPArithmetic provides all high-performance, architecture-faithful floating-point arithmetic, conversion, and rounding routines for Alpha AXP emulation. This module is used by every floating-point instruction “grain” to guarantee results and exception status that match the Alpha System Architecture (ASA), including FPCR-driven rounding, exception flagging, and full support for VAX/IEEE cross-format conversions.

 

Key Features

 

•FPCR-Driven Rounding & Exception Semantics

All operations honor the Alpha FPCR’s rounding mode and exception/trap enables. Status (inexact, overflow, underflow, invalid) is detected and reported in software; host hardware does not signal or handle traps for the guest.

 

•SIMD Acceleration, Architecture-Faithful

Where available, the module uses SSE4.1 intrinsics for scalar floating-point operations and conversions (_mm_round_ss, _mm_round_sd, _mm_cvttsd_si64, etc.), ensuring per-operation rounding and maximum performance.

 

Important:

oPer-instruction rounding is always controlled by immediate values; the host MXCSR register is never modified.

oNo reliance on host exceptions: all Alpha traps and status bits are set by the emulator, not the host CPU.

 

•VAX and IEEE Cross-Format Conversion

Includes helper routines for all conversions between IEEE (Alpha S/T) and VAX G_float, handling bitfield translation, exponent bias, and reserved operand mapping. Special cases (NaN, infinity, reserved operand) are handled per DEC/ASA rules.

 

•Header-Only, Portable, and Thread-Safe

No global state is changed. All routines are inline and safe for concurrent use on multiple threads/CPUs. The module falls back to scalar <cfenv> operations if SIMD is not available.

 

 

Design & Implementation Highlights

 

•Rounding Modes

FPCR supports round-to-nearest, toward -infinity, toward +infinity, and toward zero.

oSSE4.1 operations select rounding per instruction (never globally).

oScalar fallback uses fesetround/nearbyint per operation and restores the host rounding mode after each use.

 

•Exception Detection

All exceptions are detected in software:

oInvalid: Set when inputs are NaN, infinity, or out-of-range for INT/FP conversion (per ASA rules).

oInexact: Set when rounding changes the result (e.g., INT->FP beyond exactness threshold).

oOverflow/Underflow: Set according to IEEE/VAX rules for the operation.

oNo host CPU exception or MXCSR trap is ever relied upon; all Alpha traps are emulated.

 

VAX/IEEE Interchange

•Bitwise mapping and exponent bias adjustment for G_float <-> IEEE (S/T) conversions.

•Special value handling: reserved operand in G_float maps to IEEE quiet NaN, and vice versa.

•Conversion helpers are provided in conjunction with dt_gfloat and the exception status helpers.

 

 

Typical Usage Pattern

 

1.Arithmetic with Rounding:

•Use: FPArithmetic::round_float(value, fpcrMode)

•Use: FPArithmetic::round_double(value, fpcrMode)

•Result is rounded per Alpha FPCR, using SSE4.1 or scalar fallback.

 

2.Integer <-> FP Conversions:

•Use: FPArithmetic::int64_to_float(qint64, fpcrMode)

•Result is IEEE S with correct rounding and exception status (overflow, inexact) detected.

 

3.FP <-> Integer (Truncation):

•Use: FPArithmetic::cvt_double_to_int_chopped(double)

•Uses SSE4.1 or SSE2 truncating instructions, flags inexact/invalid per ASA.

 

4.VAX G_float <-> IEEE:

•Use: dt_gfloat and conversion helpers from OptimizedExceptionStatusRegister_Conversions.H.

 

Architecture & Reference Notes

 

Alpha System Architecture Vol. I (v6, 1994)

 

•See Chapter 4: Floating-Point Architecture for details on FPCR rounding, exception flags, and CVTxx behavior.

•CVTQS, CVTQT, CVTTQ: Integer <-> float conversion, inexact/overflow/invalid rules.

•“Floating-Point Exceptions and Traps”: Status bits and FPCR trap enables.

 

SIMD Acceleration Policy

 

•SSE4.1 is currently used only for per-operation rounding; host MXCSR/global state is never modified.

•SIMD fallback to scalar code is automatic if CPUFeatures::hasSSE41() returns false.

 

Thread Safety & Portability

 

•No shared mutable state. All routines are pure functions, header-only, and portable.

•Can be included in any Qt/C++ project targeting x86/x64 with or without SIMD.

 

 

Integration Checklist

•Ensure that every floating-point instruction grain in the emulator:

oUses FPArithmetic routines for rounding and conversion (never host default math).

oCalls conversion helpers with the correct FPCR mode and classifies exceptions using the result bundle.

oUpdates the OptimizedExceptionStatusRegister for exception flags and raises traps as required by FPCR enables.

•Always handle special values (NaN, infinity, reserved operand) per DEC/Alpha documentation, not host conventions.

•The architecture by design, hopes to deliver a standards/compliance based Alpha AXP emulator.  Where possible, and time-practicable, will incorporate optimizations (similar to SSE4 and SIMD).
