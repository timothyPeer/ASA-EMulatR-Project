PALCode & Priviledged Model Level

•PAL (Privileged Architecture Library) and supervisor/privileged mode functionality.

•These sit below the logical-level ISA and provide OS-facing low-level functionality.

•Implemented in ASA via specific PAL InstructionGrains.

 

Support for Alpha/OpenVMS, Alpha/Tru64, Alpha/Linux

PALcode (Privileged Architecture Library) is a small binary firmware written in Alpha instructions.

 

It lives in a reserved (usually ROM) memory region, loaded into the emulator at boot.

PAL runs in PALmode, with full hardware privilege — and handles:

 

•Trap & exception vectors

•TLB miss handling

•System calls (CALL_PAL)

•Machine checks

 

How PAL is Invoked:

 

Startup

•The emulator is initialized.

•AL binary is loaded into memory at its defined start address (typically 0x2000 or similar).

•PC is set to PAL reset vector (e.g., 0x2000) — because upon reset, Alpha jumps into PAL to perform platform-specific setup.

•The normal instruction loop begins at this PC.

 

Fetch–Decode–Execute

•As long as PC (program counter) points inside the PAL memory range, the fetch–decode–execute loop works exactly the same — fetching instructions from PAL binary instead of OS/user code.

•Because PAL is written in normal Alpha instructions and fully compliant with ASA, the corresponding InstructionGrain handles execution.

 

Traps into PAL

When running outside PALmode, exceptions (e.g., page fault, unaligned access) trigger a trap.

 

The emulator:

•Switches the CPU to PALmode.

•Sets PC to the appropriate PAL trap vector (inside PAL binary).

•Executes the PAL handler.

 

PAL handles the fault, then executes PAL-REI or equivalent to return to previous mode & PC.

 

Returning from PAL

PAL routines terminate by executing instructions like:

•PAL-HW_REI, PAL-REI — return from exception

•These instructions restore the previous mode & PC from saved state.

 

PC is set back to the interrupted instruction stream (OS, user code), and fetch–decode–execute continues.
