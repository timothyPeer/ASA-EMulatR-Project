Exception Frame

This page describes the architectural requirements and emulator implementation for frame fields, register storage, processor mode, and stack pointer semantics during exception and trap handling in the Alpha AXP system. Correct management of these properties is essential for architectural fidelity, OS compatibility, and proper privilege separation.

 

When a hardware trap or software exception occurs, the processor (and the emulator) must capture a complete snapshot of execution context—including all general-purpose registers (GPRs), the floating-point control register (FPCR, if enabled), and the current processor mode. These properties ensure that the interrupted process or routine can be restored with full accuracy, maintaining all privilege, access, and memory translation settings.

 

The processor mode and associated stack pointer (R30) are critical:

•The Alpha hardware saves the active processor mode and the SP as seen by the interrupted mode at the instant of exception.

•On exception return, PALcode or the OS uses these saved values to restore stack pointers (USP, KSP, ESP, SSP) and processor status, guaranteeing the resumed context runs at the correct privilege level with the expected access rights.

 

By capturing all GPRs and correctly tracking processor mode, the emulator supports:

•Reliable nested exception delivery (stack switching across K/E/S/U modes)

•OS-level context switching

•Accurate privilege and MMU state restoration

•Full compliance with Alpha architectural standards

 

This design also allows the OS or emulator to optimize for fast software exceptions (using a minimal ABI register subset) or provide full hardware compatibility (all GPRs, PS, FPCR). Floating-point registers (FPRs) are not saved in hardware frames, per architecture; software is responsible for saving them as needed.

 

The following sections detail frame field responsibilities, stack pointer restore logic, mode switching, and privilege enforcement during exception and return.

For further details, see ASA 3.2.5.2, Table 3-3, and Sections 14.2–14.3.

 

===========================

 

Frame Properties:

•m_gpr[32]: Covers all general purpose integer registers; matches the hardware frame for full architectural compliance. If you need only the ABI-preserved subset for "fast" software exceptions, you may keep the smaller set, but full hardware compatibility means all GPRs.

•m_fpcr: Only meaningful if PS.FEN is set (hardware enables floating-point).

•ProcessorMode: Required to restore stack pointer correctly (K/E/S/U).

•No FPRs: PALcode/hardware does not save FPRs on a hardware trap; software does if needed.

 

Processor Mode:

 

•When a trap or exception occurs, hardware saves the current mode (via PS) along with the stack pointer (SP, usually R30).

•To restore the interrupted process or routine, the OS/PALcode must:

oRestore the correct stack pointer for the processor mode that was active at the time of the exception (USP, KSP, ESP, SSP, depending on U/K/E/S).

oRestore processor status and privilege level from the frame, so the resumed context has the same permissions.

 

Stack Pointer Semantics

•R30 in the frame is the SP as seen by the current mode at time of exception.

•Upon return, the OS must map this value back to the correct architectural stack pointer for the restored mode:

•If resuming user mode, set the USP from R30 (frame).

•If resuming kernel mode, set the KSP from R30 (frame).

 

Privilege Level and Access Rights

•Restoring the ProcessorMode (from frame or PS) ensures:

•Correct privilege checks are re-established.

•Code resumes with the expected MMU settings, address translation, and instruction permissions.

 

Selecting Stack for Nested Exceptions

•If an exception occurs while already in K/E/S/U mode, hardware may switch stacks (e.g., from USP to KSP) based on the processor mode saved in the frame.

•PALcode uses the mode to decide where to push/pull frames for nested exception delivery.

 

Practical Implementation in Emulator

 

When restoring from ExceptionFrame:

•Set the processor’s mode register (from frame.m_processorMode or the PS field).

•Restore the correct stack pointer from frame.m_gpr[30] into the system’s USP/KSP/ESP/SSP as appropriate.

 

If your OS/emulator uses software shadow registers for SPs, update those as well.

 

When handling exceptions:

 

•Use m_processorMode to determine which stack pointer is valid for R30 in the frame.

•Update the context manager’s stack pointers as needed for the next return.
