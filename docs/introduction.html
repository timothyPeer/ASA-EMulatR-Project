Introduction

About this Effort

 

This project has been on my wish list for more than 20 years. During that time, I was fully engaged in earning a living and lacked the time and stamina to pursue this passion. Early in 2025, I made the decision to scale back “paid” work in order to focus on this long-standing goal:

 

The ASA (Alpha System Architecture) Emulator.

 

The documentation presented here should be regarded as design notes—unvetted and extemporaneous in nature. These notes are a mixture of my own work, contributions from AI partners, and excerpts from available sources. Formatting, organization, and any factual errors are solely my responsibility.

 

It is worth noting that this documentation does not fully represent the progress of the project. Compared with the current state of the coding effort, it would not be unkind to describe the documentation as woefully incomplete and lagging behind.

 

Because these notes serve as a working template describing each functional area, material for a given function is retained even if incomplete or in need of correction. Wherever possible, references to official source documentation are included, both within this collection and in the associated C++ project sources.

 

 

This is that project.

 

Project Objectives

•Build on the Qt.IO framework to ensure portability and modern development.

•Support for multiple guest Alpha processors (from 1 up to 32 CPUs per VM instance), with a goal of accurately emulating SMP systems.

•Target Alpha, VAX, Tru64/Ultrix, and Linux as guest OSes.

•Host on any OS supported by Qt.IO, including Microsoft Windows and POSIX microkernels.

•Allow user-customizable virtual machines: flexible memory models, caching architectures, and device trees.

•Emulate virtual hardware: SCSI, Fibre Channel, Q-bus, PCI, and other buses.

•Support storage via host file-based containers (e.g., .vdsk, .vtpe) and raw partition passthrough.

•Implement robust networking, leveraging Qt.IO networking for management and guest TCP/IP, with the intent to further partition and optimize cluster/network interfaces for performance.

•Maintain architectural fidelity—the emulator aims to behave exactly as the original Alpha hardware.

 

 

Strategy and Approach

 

The Learning Curve

 

I began by downloading and purchasing every Alpha architecture reference I could find—most importantly the "Alpha Architecture Reference Manual" by Sites and Witek. However, reading with comprehension was a challenge; I had no previous context of the deep internals of Alpha hardware (registers, control/status bits, I/O, memory systems).

 

Solution: I turned to a combination of “hard-copy” documentation and interactive learning with AI partners—primarily ChatGPT and, to a lesser extent, Claude.ai. My initial approach, “Trust but Verify,” failed: AI outputs were either incomplete, hallucinated, or conflicted with each other and even themselves. Reliable answers required multiple iterations, constant cross-checking, and frequent return to the primary source manuals.

 

The Reality of AI Assistance

 

AI produced results that either overreached the context of my query or were simply incomplete. I fought both extremes—misleading oversimplifications and needlessly complex “hallucinations”—in the search for a true understanding of what was actually “under the hood.” After months of topic-specific question and answer, I was finally ready to start building up the emulator piece by piece.

 

But this process wasn’t linear. My own lack of deep architectural understanding meant frequent rewrites as each piece clarified how the real hardware functioned. AI partners often produced conflicting or incorrect answers to nearly identical questions. I found I could not trust any AI-generated code without thorough manual vetting.

 

 

Adopting Standards and Patterns

 

After much iteration, I decided my design would align as closely as possible with the architecture reference manuals (Sites, Witek, et al.), and that where possible, patterns should be applied consistently across all subsystems:

 

Memory: I sketched out canonical “physical RAM” and “virtual memory” models, ensuring all memory access passes through common, auditable pipelines.

 

CPU: I implemented the Alpha CPU as a template, separating context, state, and execution logic.

 

Instruction Handling:

•Inspired by the Microsoft Orleans project, I modeled instructions as single-function “grains.”

•Each grain contains the entire logic for one of the ~700 Alpha RISC instructions.

•Grains are stateless and read-only after construction—canonical, thread-safe, reusable, and performance-optimized.

•Instructions are executed via “instantiate-once, execute-many” model: each grain is decoded once and can then be called repeatedly.

•Grains are cached (L1/L2, L3 for SMP) for efficiency, supporting high-performance instruction dispatch.

•The system architecture anticipates a future transition to native JIT (x86-64) for even higher host performance.

 

 

Technical and Architectural Challenges

1.Ambiguity in Documentation: Even “canonical” Alpha documentation (DEC/Compaq/HP) is often incomplete, ambiguous, or scattered across many sources—requiring careful reconciliation.

2.AI Code Quality: AI-generated code, while helpful for brainstorming, proved untrustworthy—every line had to be checked for accuracy and architectural compliance. For most cases, it needed to be abandoned and a standard-pattern needed to be developed or, implemented.

3.Architectural Depth: The Alpha architecture exposes every subsystem (TLBs, caches, PALcode, multiple privilege modes, SMP, complex I/O, etc.)—requiring deep dives into every hardware abstraction.

4.Emulating Real Hardware: Real Alpha hardware had quirks, errata, and device-specific behaviors, many of which are poorly or incompletely documented. Reproducing these quirks for OS compatibility is a constant challenge.

5.Performance and Scalability: SMP support, L1/L2/L3 cache models, TLB shootdowns, and atomic operations are not “optional.” These features must scale to many CPUs, remain thread-safe, and be architecturally correct.

6.Host/Guest Model Complexity: Supporting both container-backed and raw partition storage, with flexible network models, on every host OS Qt.IO supports, requires constant portability work.

7.Testing and Verification: The only true test of architectural fidelity is to run actual OpenVMS, Tru64, or Linux—which in turn requires that every subtlety (down to PALcode and firmware) is correctly handled.

 

 

Why This Matters

 

This project is about more than just emulating Alpha hardware. It’s about preserving an entire generation of computing architecture—one that was innovative, sometimes eccentric, and which remains essential for legacy software and for understanding how “real computers” work.

 

By documenting this journey—including the false starts, AI missteps, and victories—I hope to provide a roadmap for anyone else attempting to preserve classic architectures, or who simply wants to see how deep the rabbit hole goes.

 

Thank you for reading, and for any support or suggestions you can provide. The work continues.

 

Timothy Peer
