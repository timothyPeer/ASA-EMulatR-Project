Configuration Guide

The ASA is expected to be highly configurable.  Configuration settings will be configured in a .INI file and at run-time boot-load, all devices will be initialized and configured.  Devices are expected to be defined in the .INI file and attached to the running ASA-Alpha MMIO subsystem and associated sub

 

General Format and Ordering

Each line follows the pattern:

 

<Key>=<Value>

 

Sections are introduced by a line containing [SectionName]. All keys beneath that header belong to that section until a new section is declared. Sections must appear in this order (some are optional, but ordering helps with human readability):

 

; Global format comment lines (purely illustrative, and are not parsed at run-time)

 

Below each section header, parameters are documented (allowed values, units, recommendations).

 

[System] Section

Configures the overall host‐side resources allocated to the guest Alpha server.

 

[System]

; Memory size in GB (minimum 4GB)

; Typical values: 4, 8, 16, 32, 64

; Real Alpha systems supported 64MB–64GB depending on model

; Note: Larger memory sizes require more host system RAM

MemorySize=8

 

MemorySize

•Type: integer (scaled in GB)

•Description: Total guest RAM. Must be ≥ 4 (GB).

   Example: MemorySize=16 (allocates 16 GB of guest memory).

 

[CPU] Section

Defines the virtual CPU topology, cache coherency, and JIT behavior.

 

[CPU]

; Processor Count: number of guest CPUs (depends on model)

;   DS10: 1  | DS20: 1–2 | ES40/ES45/ES47: 1–4

Processor-Count=4

 

; Coherency Cache: size of the shared cache (KB) for SMP coherence

; Typical: 1024, 2048, 4096

Coherency-Cache=2048

 

; Enable Just In Time compilation (true/false)

JIT=true

 

; JIT Threshold: executions before JIT compile (10–100)

JIT-Threshold=50

 

; Instruction cache parameters (shared by all CPUs)

; Must be powers of two

InstructionCacheSize=32768

InstructionCacheLineSize=64

InstructionCacheAssociativity=4

InstructionCacheReplacement=LRU

InstructionCacheAutoPrefetch=true

 

Processor-Count

•Type: integer

•Values:

oDS10 → 1

oDS20 → 1–2

oES40/ES45/ES47 → 1–4

•Description: Number of logical Alpha CPUs to emulate.

 

•Coherency-Cache

•Type: integer (KB)

•Description: Shared cache size used for inter‐CPU coherency. Larger values reduce false sharing but increase host RAM usage.

 

•JIT

•Type: boolean (true/false)

•Description:

otrue → Enable dynamic binary translation (faster, but may have compatibility issues).

ofalse → Interpret instructions one by one (slower, but more deterministic).

 

•JIT-Threshold

oType: integer

oRange: 10–100 (executions)

oDescription: How many times an instruction block runs before it is JIT‐compiled. Lower → more frequent compilation (higher overhead); higher → fewer compilations (slower initial runs).

 

•InstructionCacheSize

oType: integer (bytes)

oRequirement: power of two, ≥ 1024

oCommon Values: 8192 (8 KB), 16384 (16 KB), 32768 (32 KB), 65536 (64 KB).

 

•InstructionCacheLineSize

oType: integer (bytes)

oRequirement: power of two

oCommon Values: 32 (EV4/EV5), 64 (EV6+), 128.

 

•InstructionCacheAssociativity

oType: integer (ways)

oRequirement: power of two

oExplanation:

▪1 → Direct mapped

▪2 → 2‐way set associative

▪4 → 4‐way set associative

▪8+ → Higher associativity (less conflict, slower lookup)

 

•InstructionCacheReplacement

oType: string

oAllowed: LRU, LFU, RANDOM, FIFO

oDescription: Policy to evict a cache line when the set is full.

 

•InstructionCacheAutoPrefetch

oType: boolean

oDescription:

▪true → Prefetch next sequential line on hits (improves streaming performance).

▪false → Only load on demand.

 

[Session-Log] Section

Controls how and where emulator logging is written.

 

 

[Session-Log]

; fName: full path for log file (Windows or Unix format)

fName=c:\asa\es40_instance.log

 

; Method: “append” or “overwrite”

;   append  → Add to existing file

;   overwrite → Replace file on startup

Method=append

 

; hw-Model: string for identifying hardware in logs

hw-Model=AlphaServer 40

 

; hw-Serial-Number: alphanumeric identifier

hw-Serial-Number=AX122312341243134

 

•fName

oType: string (file path)

oDescription: Path to the log file. Must be writable.

•Method

oType: string

oAllowed: append, overwrite

oDescription: Logging strategy at emulator launch.

 

•hw-Model

oType: string

oExamples: AlphaServer 40, AlphaServer 800, DS10, DS20

 

•hw-Serial-Number

oType: string

oNotes: 16–20 character alphanumeric for uniqueness (purely informational).

 

[ROM] Section

Specifies firmware and NVRAM files.

 

 

[ROM]

; fName: Primary ROM/firmware image (SRM console included)

fName=c:\asa\ev67.bin

 

; fName-SRM: Separate SRM console image (optional)

fName-SRM=clip67.bin

 

; Cmos-NVRam-FileName: Path for CMOS/NVRAM data (persistent)

Cmos-NVRam-FileName=clip67.dat

 

fName

•Type: string (file path)

•Description: Main ROM image that boots SRM and OS loader.

 

•fName-SRM

oType: string (file path)

oDescription: If SRM console is separated from main image. Leave blank if embedded in fName.

 

•Cmos-NVRam-FileName

oType: string (file path)

oDescription: Emulated CMOS/NVRAM storage for BIOS settings. Created if not present.

 

[Serial-Lines] Section

Defines console and auxiliary serial lines. Each “line” is identified by a name index (e.g. OPA0, OPA1).

 

[Serial-Lines]

; Format: <PortName>_<Property>=<Value>

; Primary console port OPA0

OPA0_name=OPA0

OPA0_iface=com1:

 

; Alternative or network based console (OPA1)

OPA1_name=OPA0

OPA1_iface=Net:

OPA1_net-cfg_Port=12345

OPA1_application=putty -load opa0

 

•<PortName>_name

oType: string

oExample: OPA0_name=OPA0

oDescription: Logical name of the serial line.

 

•<PortName>_iface

oType: string

oValues:

ocom1:, com2:, … (Windows COM ports)

o/dev/ttyS0, /dev/ttyUSB0, … (Linux)

oNet: (indicates network‐backed console)

 

•<PortName>_net-cfg_Port

oType: integer (TCP port)

oDescription: If iface=Net:, this TCP port listens for telnet connections.

 

•<PortName>_application

oType: string (command line)

oDescription: External application (e.g. PuTTY) to launch for console access. Use %s if you need to substitute the port.

 

You may define multiple serial lines by incrementing the index (e.g. OPA2_name=OPA2, OPA2_iface=com3:).

 

[IO-Manager] Section

Configures the I/O thread pool used by device emulation.

 

[IO-Manager]

; ThreadCnt: number of concurrent I/O threads

; Range: 1 to <# host cores>

ThreadCnt=4

 

 

ThreadCnt

oType: integer

oDescription: Number of worker threads to process asynchronous I/O (disk, network, etc.). More threads increase parallelism but consume more CPU.

 

[Network-DE500] Section

Defines Digital (DEC) DE500 Ethernet controller interfaces.  The count of devices connected is likely constrained by the PAL firmware and/or the guest OS (OpenVMS, Linux, Tru64).

 

 

[Network-DE500]

EWA0_name=EWA0

EWA0_iface=connection:Ethernet0

 

EWB0_name=EWB0

EWB0_iface=connection:Ethernet1

 

•EWA0_name, EWB0_name, …

oType: string

oDescription: Logical name for each interface (must be unique).

 

•EWA0_iface, EWB0_iface, …

oType: string

oFormat: connection:<host_interface_name>

oExamples: connection:Ethernet0, connection:br0, connection:eth0.

 

You can enumerate more interfaces as EWC0_name/EWC0_iface, etc., but for DE500, two are typical: EWA0 and EWB0.

 

[Network-DE602] Section

Configures Digital DE602 Ethernet controller (another DEC NIC type).

 

 

[Network-DE602]

EWC0_name=EWC0

EWC0_iface=connection:Ethernet2

 

•EWC0_name

oType: string

oDescription: Logical name for the first DE602 interface.

 

•EWC0_iface

oType: string

oFormat: connection:<host_interface_name>

oExample: connection:Ethernet2.

 

You may add EWC1_name/EWC1_iface if the system supports multiple DE602 interfaces.

 

[Storage-Controllers-KZPBA] Section

Describes each SCSI controller attached to the KZPBA bus. Each controller uses a two-letter code (PKA, PKB, PKC, …).

 

 

[Storage-Controllers-KZPBA]

PKA_name=PKA

PKA_scsi-id=2

 

PKB_name=PKB

PKB_scsi-id=3

 

PKC_name=PKC

PKC_scsi-id=4

 

•<Controller>_name

oType: string

oExample: PKA_name=PKA

oDescription: Logical name for the SCSI controller. Must match in other sections (e.g. DEV_KZPBA).

 

•<Controller>_scsi-id

oType: integer (0–7)

oDescription: SCSI ID on the bus (client ID). 2, 3, 4… depending on how you want to map devices.

 

You can define as many controllers as needed, using successive letters: PKA, PKB, PKC, PKD, etc.

 

[DEV_KZPBA] Section

Lists all devices (disks, tapes, CD-isos) attached to each Storage-Controllers-KZPBA entry. The count of devices connected is likely constrained by the PAL firmware and/or the guest OS (OpenVMS, Linux, Tru64). Two sub‐formats are supported:

 

1.Index‐based: <Controller>_dev[<index>]=<path>

2.Unit‐ID based: <Controller>_units[<index>]=<unitId>,<path>

 

 

[DEV_KZPBA]

; ===== PKA Controller Devices =====

PKA_dev[0]=G:\Charon\PaDS20\dka0.vdisk

PKA_dev[1]=\\.\PhysicalDrive0

PKA_dev[2]=G:\Charon\PaDS20\dka0_sys\ST136403LC.vdisk

 

; ===== PKB Controller Devices =====

PKB_dev[0]=G:\Charon\PaDS20\dka2_dka3.vdisk

 

; ===== PKC Controller Devices =====

PKC_dev[0]=G:\Charon\PaDS20\ST3146854LC_raid.vdisk

PKC_dev[100]=\\.\Tape0

PKC_dev[101]=file.iso

 

; ===== Alternative: explicit unit mapping =====

PKA_units[0]=0,G:\Charon\PaDS20\dka0.vdisk

PKA_units[1]=1,\\.\PhysicalDrive0

PKB_units[0]=0,G:\Charon\PaDS20\dka2.vdisk

PKC_units[0]=0,\\.\Tape0

 

1. Index‐based Device Lines

•<Controller>_dev[<index>]

oType: string (file path or device path)

oIndex Range: 0…n for virtual disks; 100…n for physical/tape devices

oDescription:

oIndexes 0…99 typically map to “virtual disk files” (e.g. .vdisk).

oIndexes 100+ map to physical devices or tape drives (Windows style \\.\PhysicalDriveN, \\.\TapeN).

oYou may also specify ISO images or any file format your SCSI driver supports.

 

•Examples:

oPKA_dev[0]=G:\Charon\PaDS20\dka0.vdisk

oPKA_dev[1]=\\.\PhysicalDrive0

oPKC_dev[101]=file.iso

 

The emulator will assign SCSI unit numbers sequentially starting at 0 for each index (unless overridden by units[]).

 

2. Unit‐ID Based Device Lines

•<Controller>_units[<index>]

oFormat: <unitId>,<path>

ounitId: integer (0…15) (explicit SCSI LUN/target)

opath: same as in dev[] (virtual‐disk file, physical device, etc.)

oDescription: If you need a specific LUN numbering, use this. Otherwise the emulator auto‐assigns LUN based on dev[] ordering.

•Example:

oPKA_units[0]=0,G:\Charon\PaDS20\dka0.vdisk (maps file to LUN 0)

oPKA_units[1]=1,\\.\PhysicalDrive0 (maps drive to LUN 1)

oIf both dev[] and units[] entries exist for the same controller, units[] takes precedence for assigning LUNs.

1.Performance Tuning Guidelines

1.High Performance (typical recommendation):

2.JIT=true

3.JIT-Threshold=50

4.InstructionCacheSize=32768 (32 KB) or 65536 (64 KB)

5.InstructionCacheLineSize=64 (matches EV6+)

6.InstructionCacheAssociativity=4

7.InstructionCacheReplacement=LRU

8.InstructionCacheAutoPrefetch=true

9.ThreadCnt=4 (I/O threads)

10.Guest memory ≥ host RAM / 2 (e.g. if host has 16 GB, allocate 8 GB to guest)

 

2.Maximum Compatibility (older OSes or debugging):

11.JIT=false (interpretation mode)

12.Use historically accurate cache sizes:

oEV4/EV5 → InstructionCacheSize=8192, InstructionCacheLineSize=32, associativity=1 or 2

13.InstructionCacheAutoPrefetch=false (to see exact memory access patterns)

14.ThreadCnt=1 (simplest I/O ordering)

 

3.Development/Debugging Mode:

•Method=append (preserve previous logs)

•JIT=false (deterministic, easier to step through)

•MemorySize=4 (faster startup)

•Single CPU (Processor-Count=1)

•Use virtual disks (*.vdisk) instead of physical devices to avoid host OS side‐effects - To reduce risk, will post registry changes to prevent accidential initializations in a Windows Server environment.

 

Example Minimal Configuration

Below is a trimmed‐down example suitable:

 

 

[System]

MemorySize=4

 

[CPU]

Processor-Count=1

Coherency-Cache=1024

JIT=false

JIT-Threshold=50

InstructionCacheSize=8192

InstructionCacheLineSize=32

InstructionCacheAssociativity=1

InstructionCacheReplacement=LRU

InstructionCacheAutoPrefetch=false

 

[Session-Log]

fName=/tmp/alpha.log

Method=overwrite

hw-Model=DS10

hw-Serial-Number=DS10TEST12345678

 

[ROM]

fName=./ev4.bin

fName-SRM=

Cmos-NVRam-FileName=./nvram.dat

 

[Serial-Lines]

OPA0_name=OPA0

OPA0_iface=/dev/ttyS0

 

[IO-Manager]

ThreadCnt=1

 

[Network-DE500]

EWA0_name=EWA0

EWA0_iface=connection:eth0

 

[Storage-Controllers-KZPBA]

PKA_name=PKA

PKA_scsi-id=2

 

[DEV_KZPBA]

PKA_dev[0]=./dka0.vdisk

 

Adjust paths, models, and sizes as needed. This configuration boots a single‐CPU DS10 with 4 GB RAM, an 8 KB direct‐mapped I‐cache, no JIT, one I/O thread, a single serial console, and one virtual disk.
