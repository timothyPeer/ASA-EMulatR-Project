TLB Exception Handling in the ASA-EMulatR

Translation Lookaside Buffer (TLB) exceptions occur when the processor attempts to translate a virtual address that is either not present in the TLB (miss), is invalid, or does not meet the required access permissions. TLB exceptions are fundamental to Alpha AXP memory management and are handled as part of the processor’s exception and interrupt subsystem.

 

How TLB Exceptions are Detected and Handled:

 

Detection:

•During instruction execution, a memory access (load, store, fetch) triggers a virtual-to-physical address translation.

•The TLB is consulted. If there is no valid matching entry (miss), or if the entry is invalid or permission-check fails, a TLB exception is signaled.

 

Exception Notification:

•The exception is raised by the memory subsystem or by the instruction grain (e.g., on a load/store).

•The grain or memory subsystem calls AlphaCPU::RaiseException, passing the exception category (typically MEMORY_FAULT) and the specific reason (TLB_MISS, PAGE_FAULT, or ACCESS_VIOLATION).

•The exception detail includes the faulting virtual address, access type, and any relevant TLB state.

 

Context Save and Trap Vectoring:

•AlphaCPU saves the current processor context, including the PC, privilege state, and (optionally) the virtual address and instruction type.

•A hardware-style trap frame (e.g., frameTLBException or HWStackFrame) is constructed and pushed to the exception stack.

•Control is transferred (vectored) to the appropriate PALcode handler (such as the TLB miss or page fault handler).

 

PALcode/OS Handling:

•PALcode or the OS inspects the exception details, examines the cause, and attempts to resolve the fault (e.g., by loading a page table entry into the TLB, updating access permissions, or terminating the process for a true access violation).

•The handler may update TLB entries, reset the processor state, and resume execution at the faulting or next instruction as appropriate.

 

Status and Exception Recording:

•The OptimizedExceptionStatusRegister may be updated to indicate a memory or TLB fault.

•Relevant IPRs (Internal Processor Registers), such as EXC_ADDR (faulting address), are updated for handler inspection.

 

Notes:

•TLB exceptions are typically precise exceptions: the processor state at the moment of the fault is saved exactly, so software can retry or diagnose the instruction.

•Unlike arithmetic exceptions, TLB exceptions almost always result in an immediate trap; “sticky” status bits are less common, but diagnostic flags may still be set.

•There is no architectural "TLB Exception Frame" in Alpha AXP; TLB exceptions use the standard hardware interrupt/trap frame, with additional software-specific data for diagnostics if desired.
