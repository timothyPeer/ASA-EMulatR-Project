Conversion Helper Functions

Scope and goals

 

This chapter standardizes how conversions set IEEE/VAX floating-point exception semantics, feed the emulator’s status/trap machinery, and surface rounding results. The helpers are designed to be:

 

•Deterministic and architecture-faithful

•Header-only, noexcept, and usable from instruction grains without allocations

•FPCR-aware (rounding mode, trap enables, and status flags)

•Suitable for both “check only” (flagging) and “compute & pack” (produce final value) call sites

 

The functions are organized around common Alpha/VAX conversion flows that occur in CVTxx instructions and in integer<->FP boundaries. They compute a canonical result bundle that your instruction grains can consume to set flags, raise traps (if enabled), and write back the converted value.

 

Background: architectural rules that drive conversion

 

Alpha AXP floating-point behavior is IEEE-754 conformant for S and T formats with Alpha-specific FPCR and exception semantics. VAX formats (notably G_float) follow VAX rules for exponent biasing, encoded fraction layout, reserved operands, and mapping to IEEE values during conversion.

 

Key behavioral anchors used by these helpers:

 

1.Rounding modes

a.Alpha FPCR supports rounding modes compatible with IEEE roundTiesToEven and directed modes. For CVTxx conversions, honor the active FPCR rounding mode unless the instruction variant prescribes an override.

b.ASA reference: Floating-Point Control Register (FPCR) — rounding mode field; FP exception model. See ASA Vol. I, Floating-Point Architecture sections (rounding and exceptions).

2.Inexact, overflow, underflow, invalid

a.Conversions can set these flags. For example, FP->INT can be invalid (NaN, infinity, or magnitude out of range), and INT->FP can be inexact when the integer magnitude cannot be represented exactly in the destination FP format.

b.// ASA reference: Exception flag behavior for conversions; integer conversion invalid rules; NaN propagation.

3.NaN, Infinity, Zero, Subnormal handling

a.IEEE S/T: preserve NaN payload sign and quietness where required; infinities out of range for integer conversions set invalid; subnormals can trigger underflow/inexact.

b.VAX G_float: “reserved operand” encodes exceptional value; map reserved operand to IEEE NaN on G->IEEE, and map IEEE NaN to a reasonable VAX reserved operand encoding on IEEE->G.

c.ASA reference: IEEE NaN/infinity mapping; VAX reserved operand rules (Digital/VAX FP specs).

4.VAX G_float particulars

a.G_float: 64-bit, exponent bias 1024, different fraction bit ordering vs. IEEE double. Conversions must reorder bits, adjust bias, and detect reserved operands correctly.

b.VAX G_float definition: exponent bias, fraction layout, and reserved operand identification.

 

Summary

 

The conversion helpers defined in this chapter give the emulator a single, consistent place to apply Alpha/VAX rules for rounding and exception classification across INT<->FP and IEEE<->VAX boundaries. Using these helpers from instruction grains ensures that exception flags and traps are set correctly, results are rounded as the ISA dictates, and special values (NaN, Inf, reserved operands) are handled faithfully.
