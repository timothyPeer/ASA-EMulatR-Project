AlphaCPU

The ASA-Alpha - AlphaCPU:: represents one virtual CPU core in the Alpha AXP emulator. It models the execution of instructions, manages per-core state, handles traps & interrupts, and interacts with memory, caches, and devices — exactly as a real Alpha CPU would.

 

Core Responsibilities:

1️. Instruction Execution

•Implements the fetch → decode → execute → commit loop.

•Fetches the next instruction word from memory (via caches & MMU).

•Resolves the instruction using GrainResolver → InstructionGrain.

•Executes the instruction logic (arithmetic, load/store, branch, PALcode, etc.).

•Advances the Program Counter (PC).

 

2️. Register State

•Owns or interacts with AlphaProcessorContext, which contains:

•General-purpose integer registers (GPRs)

•Floating-point registers (FPRs)

•Vector/SIMD registers (if implemented)

•Processor status registers (PS, FPCR)

•IPRs (internal processor registers)

 

3️. Exception & Trap Handling

Detects and responds to:

 

•Page faults

•Protection violations

•Arithmetic exceptions (overflow, divide by zero)

•Privilege violations

•Interrupts & IPIs

•Raises ExceptionInfo and pushes exception frames on the stack (StackManager).

•Coordinates with PALcode for certain traps.

 

4️. Inter-Processor Communication

•Sends and receives Interprocessor Interrupts (IPIs) through AlphaSMPManager.

•Participates in TLB shootdowns, reschedule requests, or halt signals from other CPUs.

 

5️. Memory Operations

•Works with MMU & AlphaMemorySystem to read/write memory:

•Issues virtual memory accesses, handled by MMU/TLB → physical memory.

•Detects and handles alignment faults, cache misses, MMIO accesses.

 

6️. Caches & Coherency

•Interacts with the UnifiedDataCache and UnifiedInstructionCache:

•Issues loads/stores that hit/miss in cache.

•Observes memory barriers & cache flushes as required for SMP coherency.

 

7️. Profiling & Cycle Accounting

•Tracks per-core instruction counts, cycles executed, and stall cycles.

•Updates execution profiles & timing for debugging or performance analysis.

 

How it fits in the ASA-Alpha emulator. 

 

GrainResolver → InstructionGrain → AlphaCPU (per-core)

        ↑                  |

  StackManager      AlphaProcessorContext

        ↑                  |

AlphaSMPManager → AlphaMemorySystem → MMU & Caches → Devices

    │

    ├── Translation Buffer Table (TLB) (per CPU)

    │   ├── ITLB (instruction translation cache)

    │   ├── DTLB (data translation cache) 

    │   └── Translation logic (MMU emulation)

    │

    ├── JITCompiler  (per CPU)

    │

    └── References shared AlphaMemorySystem

        ├── SafeMemory (physical RAM)

•In a symmetric multiprocessing (SMP) system configuration emulating the Alpha system architecture, the number of copies of RegisterBank and FPRegisterBank maintained in the emulator corresponds to the number of virtual CPUs (vCPUs) being emulated. Each vCPU requires its own set of integer and floating-point registers to accurately simulate the behavior of a physical Alpha processor.

•For instance, if you're emulating a 4-way SMP Alpha system, the emulator would maintain four separate instances of both RegisterBank and FPRegisterBank. This design ensures that each virtual CPU operates independently, with its own register state, mirroring the architecture of a real SMP system.

•This approach is standard in emulators like QEMU, which emulate multiple CPUs by instantiating separate CPU state structures for each vCPU. These structures include the general-purpose and floating-point registers, among other components, to faithfully replicate the behavior of each processor in the SMP system.

Considerations:

 

All memory operations (including instruction fetch) should pass through AlphaMemorySystem:

This ensures:

1.Protection bits are enforced

2.Instruction fetches don't hit MMIO

3.Faults are trapped
