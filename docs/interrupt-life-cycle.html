Interrupt Life-Cycle

The ASA-EmulatR processes interrupts using the following general flow:

 

External device/timer/etc raises interrupt

InterruptController::PostInterrupt(InterruptLevel::IPL_CLOCK, TIMER_VECTOR, context)

↓

Controller queues interrupt in QMultiHash

interrupt_queue.insert(priority_key, request)

↓

If interrupt can be delivered (IPL > current IPL)

SignalInterruptPending()

↓

Sets IP bit and emits signal

emit interruptPending()

↓

CPU receives signal and checks for deliverable interrupts

 

then -

 

1. Device raises interrupt

timer->timeout() → PostInterrupt(IPL_CLOCK, TIMER_VECTOR, nullptr)

 

2. Controller signals CPU  

SignalInterruptPending() → emit interruptPending()

 

3. CPU services interrupt

AlphaCpu::onInterruptPending() → deliverInterrupt()

 

4. Software services interrupt

ISR runs → RTI instruction

 

5. CPU returns from interrupt  

executeRTI() → SignalInterruptCheck()

 

6. Cycle repeats for any newly deliverable interrupts

Key Data Flows / APIs:

•MMIO/Device (sigInterruptRequest) --> InterruptFabric::connectDevice()

•--> IRQController::postDeviceInterrupt() --> Enqueue/dispatch

•System Timer (QTimer) --> IRQController::processInterruptTick()

•IPI (sigIPI) --> IRQController::postIPI() --> sigDeliverIPI()

•AlphaCPU[ ]: slot for handleInterrupt(), handleIPI(), checkInterruptDelivery()

•PAL: AlphaCPU invokes PAL entrypoints on interrupt delivery (see §6.5)

• Machine Check/PowerFail: IRQController::postMachineCheck() (IPL 31/30)

 

Dispatch Process (Simplified):

 

1. Device, MMIO, or software posts InterruptRequest to IRQController (via signals/slots).

2. IRQController enqueues InterruptRequest by (dstCpu, IPL).

3. On each timer tick (QTimer), IRQController emits sigDeliverInterruptToCpu() for deliverable requests (respects IPL, IE).

4. AlphaCPU receives and calls handleInterrupt(), which triggers PAL entry (interrupt vectoring and context switch as per ASA §6.5).

5. For IPI, sigDeliverIPI() directly calls AlphaCPU::handleIPI().

6. On interrupt completion, stats/latency recorded, pending bits updated.

 

PAL Integration Points:

•Software interrupts: CALL_PAL MTPR_SIRR (Alpha AXP SRM §6.4.1.2)

•IPI: CALL_PAL MTPR_IPIR (SRM §6.4.5.1)

•Machine Check: Delivered via PAL mechanism (§6.2.3, §6.5)

• The clock will execute from the AlphaSMPLauncher Class.  HW clock interrupts will emit to all CPUs at regular intervals

•All device-generated interrupts (MMIO, etc) are connected to InterruptFabric, which posts them into the IRQController.

•IRQController centrally manages all priorities (IPL), queues, and   per-CPU interrupt pending bits.

•Inter-CPU IPIs are modeled as a signal: sigIPI() emitted from any CPU or MMIO logic, delivered directly (at IPL 22) to the target AlphaCPU.

oAlphaCPU::handleInterrupt() and ::handleIPI() execute PALcode interrupt vectoring per ASA (§6.5).

•Software and timer interrupts are posted by MMIO or PAL using the same wiring.

•Machine checks and power fail are posted at highest IPLs (30, 31).

•All delivery to CPU happens by Qt signal/slot.

 

Legend:

[ ]   = object/class/module

--->  = Qt signal/slot or direct API call

PAL   = entry to PAL handler (emulator models this in AlphaCPU/PAL subsystem)

MMIO  = any device mapped to memory I/O (e.g. SCSI, NIC, Timer)
