Low-Order Address Bits and Length-of-Access Information

When the Alpha documentation mentions "sufficient low-order address bits and length-of-access information to devices on I/O buses," it's referring to important hardware implementation details that affect system integration:

 

Low-Order Address Bits

1.Complete Address Transmission: When the CPU performs memory operations like LDQ that might target I/O devices (memory-mapped I/O), the complete address must be transmitted to the I/O bus, including all low-order bits - even those that might be ignored for alignment purposes.

2.Byte-Level Granularity: While LDQ requires 8-byte alignment, I/O devices often need to know the exact byte address for proper operation. For example, a device might check specific address bits to determine which internal register is being accessed.

3.Implementation Requirement: Alpha implementations must ensure that even though LDQ operates on 8-byte boundaries (with the lowest 3 bits effectively ignored for memory access), those bits are still transmitted to the I/O bus rather than being masked off internally.

 

Length-of-Access Information

1.Size Signaling: When an LDQ instruction accesses memory, the memory controller or I/O device needs to know that this is an 8-byte (quadword) operation.

2.Bus Protocol: The Alpha bus protocols must communicate not just the address but also the size of the operation (1, 2, 4, or 8 bytes).

3.Partial Operations: Some I/O devices might handle differently sized accesses in different ways, so knowing that an LDQ is requesting 8 bytes is critical.

Floating-Point Condition Code - Used to record results of floating-point comparisons for branching and status inspection.

FPCR

Floating-Point Control Register - The FPCR is a privileged processor register in Alpha AXP (and similar RISC architectures) that controls and reports the state of the floating-point unit (FPU).

 

It contains control bits (such as rounding modes and trap enables), status/sticky bits (indicating if exceptions have occurred), and floating-point condition codes (FPCC), among others.

•The FPCR directly determines how the processor handles floating-point exceptions (trap vs. sticky), rounding behavior, and floating-point branching.

•Key Fields in FPCR (Alpha AXP, Table 9–1):

oTrap Enable Bits: Enable/disable trapping for individual FP exceptions (invalid, overflow, underflow, divide-by-zero, inexact).

oStatus/Sticky Bits: Indicate if a corresponding FP exception has occurred (“sticky” until cleared by software).

•Rounding Mode: Controls FP rounding (nearest, plus infinity, minus infinity, chopped).

•FP Condition Codes (FPCC): Result of last FP comparison (EQ, LT, GT).

 

Other: Dynamic mode, summary, reserved bits, implementation-specific fields.
