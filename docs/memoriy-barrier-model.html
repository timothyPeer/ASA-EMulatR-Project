Memory Barrier Model

Alpha Architecture Memory Barrier Model

The Alpha architecture implements a weak memory consistency model with explicit memory barriers. In this model:

•Loads and stores may be reordered by the CPU

•Memory barriers (MB, WMB) enforce ordering constraints

•Load-Locked/Store-Conditional pairs provide atomic operations

 

For SMP systems, proper synchronization requires:

1.Reservations tracked per CPU

2.Cache line invalidations across CPUs

3.Proper barrier instructions

 

Memory Barrier Instructions

 

MB : Full memory barrier, enforces ordering of all memory operations

WMB : Write memory barrier, enforces ordering of all write operations

 

Implementation Details

Reservation Mechanism

Each CPU maintains a reservation state consisting of:

 

Address: The virtual address that was locked

Size: The size of the locked operation (4 or 8 bytes)

Valid Flag: Whether the reservation is still valid

 

Alignment Requirements

 

LDL_L - Load Locked Longword / STL_C - Store-Conditional Longword : Must be 4-byte aligned

LDL_Q - Load Locked Quadword / STQ_C - Store-Conditional Quadword : Must be 8-byte aligned

 

Misaligned accesses result in:

 

For Load-Locked: Alignment fault

For Store-Conditional: Automatic failure (returns 0)

 

Fault Handling

Operations may generate various faults:

 

•Alignment Fault: Access not properly aligned

•Access Violation: Insufficient permissions

•Translation Not Valid: Virtual address not mapped

 

Reservation Invalidation Causes

A reservation becomes invalid when:

 

•Any CPU writes to the same cache line

•The same CPU executes another Load-Locked instruction

•The CPU executes a Store-Conditional instruction

•A memory barrier (MB) is executed

•An exception is taken

 

SMP Coherency Model

The SMP implementation ensures proper cache coherency through a centralized lock manager in the AlphaSMPManager class.

 

Lock Tracking

The following data structures track locks across the system:

 

•m_cpuLockReservations: Maps CPU IDs to their locked cache lines

•m_cacheLinesWithLocks: Maps cache lines to the set of CPUs that have locked them

•m_invalidatedCacheLines: Set of cache lines that have been written to since locks were established

 

Cross-CPU Invalidation Flow

 

1.CPU A executes LDL_L or LDQ_L at address X

2.AlphaJITCompiler records the reservation and notifies AlphaSMPManager

3.AlphaSMPManager records that CPU A has a reservation on the cache line containing X

4.If CPU B writes to the same cache line:

a.AlphaSMPManager::handleMemoryWrite is called

b.The cache line is marked as invalidated

c.CPU A's reservation is invalidated

 

5.When CPU A executes STL_C or STQ_C:

a.AlphaJITCompiler checks local reservation validity

b.It then queries AlphaSMPManager to verify no other CPU has written to the cache line

c.If valid, the store succeeds and returns 1; otherwise, it fails and returns 0

 

 

 

Cache Line Granularity

The implementation assumes 64-byte cache lines (common in Alpha architectures). This means:

 

•Writing to any address within a 64-byte block will invalidate reservations on any address in that block

•This matches hardware behavior, where cache coherency operates at cache line granularity

 

 

Example: 

// Alpha assembly for atomic increment of a memory location

// Assuming address is in $16

 

loop:

    LDL_L   $0, 0($16)     // Load current value with lock

    ADDL    $0, 1, $1      // Increment value

    STL_C   $1, 0($16)     // Try to store new value

    BEQ     $1, loop       // If failed (register $1 == 0), retry

 

Emulated execution flow:

 

1.CPU executes LDL_L and establishes a reservation

a.m_lockReservationAddr set to address

b.m_lockValid set to true

c.m_lockReservationSize set to 4

d.AlphaSMPManager::notifyLockReservation called

 

2.CPU executes ADDL (normal integer operation)

a.CPU executes STL_C

b.Check if reservation is valid locally

c.Check with AlphaSMPManager::checkLockReservationValid

d.If valid, perform store and return 1 in the destination register

e.If invalid, return 0

f.Always clear m_lockValid

3.Branch if store failed, creating a retry loop
