Alpha 754 double-precision / Alpha T_floating (64-bit/128) format)

A robust, nearly ready for production, strict IEEE conformance, clarity, and edge-case handling.  The dt_tfloat implements the Alpha AXP T_Float (quad precision), a 128-bit floating-point type (IEEE 754 binary128/quadruple), modeled for accurate Alpha emulation. It encapsulates two 64-bit words, with a full register-level interface and conversion helpers.

 

File: dt_tfloat.h

 

 

Field Layout (IEEE 754 Quad Precision)

| 127 | 126        112 | 111                                0 |

+-----+----------------+--------------------------------------+

|  S  |   Exponent     |              Fraction                |

| (1) |   (15 bits)    |               (112 bits)             |

+-----+----------------+--------------------------------------+

•S: Sign bit

•Exponent: 15 bits (bias 16383)

•Fraction: 112 bits (MSB implicit as in IEEE)

 

Data are stored as two words:

•m_high – bits 127:64

•m_low – bits 63:0

 

1. General Design

•Clean dual-mode layout: m_low = T_floating or lower 64 bits of quad, m_high = upper 64 for quad, always 0 for T_floating.

•Constructors, copy, assignment all correct.

•Conversion methods for all major scalar and float types.

•Factory and accessor functions are consistent with Qt/C++ idioms.

 

2. IEEE Compliance and T_floating Details

•T_floating (64-bit)

•Follows Alpha AXP Table 4-9: Sign (63), Exp (62:52, bias 1024), Fraction (51:0).

•Handles exponent bias conversion with IEEE double.

•Handles subnormal, zero, infinity, and NaN correctly (see notes below).

 

   IEEE quad (128-bit)

•Follows IEEE 754-2008: Sign (127), Exp (126:112, bias 16383), Fraction (111:0).

•Handles endian issues with swap.

 

3. Conversion and Edge Cases

 

fromFloat() and fromDouble():

•Properly converts subnormal/zero to T_floating zero.

oInf/NaN handled with helper, but for NaN, you use a canonical encoding (0x7FF8000000000000ULL).

•This is valid as a default quiet NaN; could expand to distinguish sNaN vs qNaN.

•Overflow: The mapped T_floating exponent overflows (exp_t >= 2047), set to Inf.

 

fromLongDouble() and fromIEEEQuad():

•Use of endian swap and memory copy.

•On some platforms, long double is not quad (128b). A fallback handles this case.

 

fromInt64/fromUInt64:

•Handles sign and bias, includes correct rounding to fraction.

•Finds the MSB to calculate unbiased exponent (standard technique).

oGood use of rounding.

 

toDouble(), toFloat():

•T_floating exponent <-> IEEE handled by bias diff (±1).

•Subnormals mapped to zero.

•Inf/NaN preserved for IEEE double.

 

isNaN(), isInf():

•Both quad and T_floating handled per IEEE and Alpha conventions.

oChecks for nonzero fraction for NaN, zero for Inf.

 

isFinite(), isSignBitSet():

•Simple and accurate.

 

Operator==/!=, reset, accessors:

•Equal&Not-Equality Override accessors.

 

4. Special Value Creation

•createSpecialValue(double value, FloatFormat format) is clean and maps ±Inf/NaN for both T_floating and quad.

oAlways generate a quiet NaN (0x7FF8...).

•If signaling NaN, use a fraction with MSB zero (see IEEE 754).

 

5. toIEEEQuad_as_double()

 

•Extract sign, exp, and frac from the upper 64 bits for quad; this is correct for most little-endian quads.

•Correctly handles underflow/overflow.

•Maps only top 52 bits of the quad fraction to double (the best possible with loss).

 

6. Improvements/Edge Case Suggestions

 

fromFloat() and fromDouble():

•When mapping subnormals, set to zero. Alpha T_floating does not use denormals—mapping to zero is architecturally compliant.

 

fromInt64():

•The fraction calculation is robust.

 

Endian handling for IEEE quad:

•If targeting both big/little-endian hosts, always clarify which memory layout is canonical.

 

Key Methods and Conversions

Construction

•dt_tfloat() – zero

•dt_tfloat(quint64 low, quint64 high) – explicit

•Conversion from various smaller types: fromFloat, fromDouble, fromLongDouble, fromInt32, fromInt64, etc.

•static dt_tfloat fromFloat128(double upper, double lower = 0.0)

Accepts two doubles (for quad support; may require platform/compiler to actually use IEEE quad math).

 

Conversion to Native Types

 

•float toFloat() const – uses low 32 bits

•double toDouble() const – uses low 64 bits

•long double toLongDouble() const – full 128 bits if supported

•Various toInt32, toUInt64, etc.

 

Data/Range Utilities

•bool isZero() const – checks if both words are zero

•bool isNaN() const, bool isInf() const – via double conversion

•Bit tests: hasHighWord(), fitsIn64Bits(), etc.

•Arithmetic (128-bit multiply)

•static dt_tfloat multiplySigned(qint64 a, qint64 b)

Returns a 128-bit signed multiply (for instructions like MULQ).

 

Status/Property Queries

•bool isReservedOperand() const

•True if exponent == 0x7FFF and fraction != 0 (IEEE quad NaN/reserved operand).

•Reference: IEEE 754-2008, ASA Vol I Table 2-7.

•bool isOverflow() const

oExponent == 0x7FFF (all ones)

•bool isUnderflow() const

oExponent == 0, fraction != 0 (subnormal/denormal)

•bool isExactResult(long double a, long double b, long double result) const

True if result, when recoded as T_Float, exactly matches this value.

 

Raw Access

•quint64 low64() const, quint64 high64() const

•For register/memory writing

•void setLow64(quint64), void setHigh64(quint64)

 

Display

•QString toHexString() const

•Printable form (high:low)

 

Example Usage & Architectural Notes

•G_Float is used to model VAX/Alpha VMS compatibility. For "real" IEEE math, prefer S, T, or D float unless targeting VAX-legacy operations.

•T_Float (quad) is required for certain Alpha opcodes (see: ADTT, SUBT, etc.), for accurate trap, overflow, and inexact detection, especially for scientific and emulation contexts.

 

When writing instruction grains:

 

Always use the type matching the target Alpha opcode:

 

•dt_sfloat for S_floating (32b),

•dt_dfloat for D_floating (VAX D),

•dt_gfloat for VAX G,

•dt_tfloat for Alpha T_floating (quad).

 

For correct overflow, underflow, reserved operand, and inexact checking, always use the provided methods (not just host math), as these types encode Alpha/VAX architectural quirks (exponent bias, reserved bits, endianness).
