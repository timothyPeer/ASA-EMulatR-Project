OptimizedExceptionStatusRegister

 

AlphaCPU owns and orchestrates instruction execution, traps, pipeline flow, and all architectural side effects.

•By storing OptimizedExceptionStatusRegister in AlphaCPU, all exception state—including ESR bits, floating-point condition flags, and the full context for trapping—is centralized.

•All traps and exception-raising routines naturally live here, so having the bitfield available makes diagnostic and dispatch logic seamless.

 

Architectural Accuracy

•In real hardware: The ESR and FPCR are per-CPU architectural registers (or per-processor context), not per-instruction.

•Software (PALcode, OS) expects to read these as part of the CPU state when traps or faults are delivered.

 

Single Source of Truth

Exception bits may be set during instruction execution, but their effects (e.g., trap, summary bits, clearing, masking, etc.) are only recognized and acted on by the CPU control logic.

For example:

•On an exception, ESR/FPCR/EXC_SUM and related IPRs are set/updated

•Trap vector is computed

•Context is saved and trap handler is entered—all by AlphaCPU

 

Simplifies Handler/Trap Dispatch

 

When a trap or exception is raised (e.g., via AlphaCPU::raiseException, raiseMemoryException, signalIntegerOverflowTrap, etc.), the logic can:

 

•Read and set bits in OptimizedExceptionStatusRegister

•Populate IPRs (EXC_SUM, EXC_ADDR, EXC_PC, etc.)

•Push trap frames, update mode, change PC

•Clear or mask flags after trap (reset ESR as needed)

•All of these steps are local to AlphaCPU.
