Introduction to Instruction Grains

The AlphaInstructionGrain is the fundamental building block of the ASA-EMulatR’s instruction execution model. It provides a lightweight, stateless abstraction for representing a single Alpha AXP instruction in a canonical form, independent of its opcode or function code specifics. Each instruction in the Alpha ISA is mapped to a derived Grain class, where decoding, validation, and execution logic is encapsulated.

 

By standardizing all instructions under a common base class (AlphaInstructionGrain), we gain uniformity in instruction dispatch, exception handling, and architectural validation, while preserving modularity and performance.

 

GRAIN CONCEPT

The term "Grain" refers to the smallest indivisible unit of instruction execution.

 

Each instruction grain implements:

 

• decode(): Translates the raw 32/64-bit instruction word into its architectural fields (opcode, function, register indices, displacement).

•execute(): Executes the instruction semantics against the AlphaProcessorContext and AlphaCPU, including arithmetic, memory access, or control transfer.

•opcodeAndFunction(): Provides a canonical identifier tuple (opcode, function) for table-driven dispatch.

•mnemonic(): Returns the human-readable mnemonic string (e.g., "ADDQ/V").

•grainPlatform(): Declares applicability (Alpha, VAX, Tru64, Unix, etc.).

 

Because all grains derive from the same base, the CPU’s dispatch loop may uniformly decode and execute instructions without embedding opcode-specific logic directly.

 

STATELESSNESS

Instruction grains are designed to be stateless objects. All runtime state is carried in the AlphaProcessorContext (register file, FPCR, PS, MMU state) and AlphaCPU (per-CPU execution state). This ensures:

• Reusability of grain instances without reallocation.

• Simplified thread-safety in SMP environments.

• Clear separation between execution logic and architectural state.

 

Stateless design also makes JIT translation easier, as grains can be transformed into code fragments without concern for hidden state.

 

DISPATCHING AND EXECUTION

The emulator constructs a dispatch table keyed by (opcode, function). Each entry points to the corresponding AlphaInstructionGrain subclass.

 

Execution flow:

  1. Fetch instruction word from memory.

  2. Lookup Grain in dispatch table using opcode/function tuple.

  3. Call grain->decode(rawInstr, cpu).

  4. Call grain->execute(cpu, context, rawInstr).

  5. Advance PC and handle any raised exceptions.

 

This model ensures O(1) dispatch time and supports clean separation of concerns.

 

EXCEPTION HANDLING

The base AlphaInstructionGrain integrates central validation hooks, conforming to ASA Chapter 14. Derived grains invoke these checks to raise precise exceptions (e.g., integer overflow, FP invalid operation, privilege violation). Exceptions are consistently routed through AlphaCPU::raiseException(), ensuring architectural compliance across all instruction types.

 

BENEFITS

 

• Modularity: Each instruction encapsulated in its own file/class.

• Scalability: Easy to add new instructions (future Alpha, PALcode, VAX).

• Consistency: Centralized validation logic ensures architectural fidelity.

• Performance: Table-driven dispatch supports fast interpretation and JIT.

• Debuggability: Mnemonics and execution hooks improve tracing and diagnostics.

• Portability: Grains can be mapped to multiple platforms (Alpha, Tru64,  OpenVMS, Unix) through ProcessorArchModes flags.

 

CONCLUSION

 

AlphaInstructionGrain provides the unifying abstraction that allows the emulator to model Alpha AXP execution faithfully while remaining modular and performant. It is the cornerstone of the grain-based execution system, bridging between raw instructions, architectural state, and exception handling.
