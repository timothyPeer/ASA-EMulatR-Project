Trap Handling

This page documents the three canonical patterns for delivering floating-point exceptions and traps in the ASA-EMulatR, as required by the Alpha AXP architecture. Every floating-point arithmetic instruction grain (ADD, SUB, MUL, DIV, SQRT, etc.) selects one of these models based on its variant suffix (e.g., /C, /D, /M, /U, etc.).

 

Each pattern defines when an exception results in a trap (architectural interrupt), and when it is only recorded as a “sticky” status bit for later inspection.

The trap delivery logic is determined by the combination of the OptimizedExceptionStatusRegister (which records whether an exception occurred) and the FPCR trap enable bits (which control whether that exception is eligible to interrupt the processor).

 

Trap Only if Both:

•The most common model for Alpha floating-point instructions (variants: /C, /D, /SU, /SUI, /SUIC, /SUD, /SUID, /SUIM, and default/none).

•The grain raises a trap only if the relevant exception status bit is set and the corresponding FPCR trap enable bit is set.

 

Trap Always:

•Used for “unmasked” or unconditional variants (e.g., /U, /UM, /UC, /UD).

•The grain always delivers a trap if the exception occurs, regardless of FPCR trap enables.

 

Status Only:

•Used for summary-only or masked variants (e.g., /M, /SUM, /SUID).

•The grain updates the OptimizedExceptionStatusRegister for any exceptions, but never delivers a trap—these exceptions are only sticky for software inspection.

 

The included code snippets illustrate each model and should be used directly in all arithmetic grains.

AlphaCpu::raiseException() is the canonical entry point for trap delivery: all architectural traps must be funneled through this method for correct context, state, and handler vectoring.

 

For implementation:

 

•Always update OptimizedExceptionStatusRegister status bits after the arithmetic operation.

•Slect and apply the correct trap pattern for the grain type based on the variant table.

•Only call AlphaCpu::raiseException() if required by the model; otherwise, update status only.

•Always write results, notify, and advance PC after exception/trap handling.

 

This approach guarantees full architectural compliance for OS, PALcode, and guest application compatibility.

 

Examples and Usage:

 

Trap Only if Both

 

(OptimizedExceptionStatusRegister + FPCR Trap Enable Checked)

Used for: /C, /D, /SU, /SUI, /SUIC, /SUD, /SUID, /SUIM, (none)

 

// Trap only if both exception occurred AND trap enable is set in FPCR

if (status->hasInvalidOpOccurred() && fpcr->hasFlag(ExceptionCategory::TRAP_INVALID_OP_ENABLE)) {

ExceptionInfo info;

quint32 bitmask = ALPHA_EXCEPTION_SRC_BITMASK(m_ra, m_rb);

info.setSoftwareIPLBitmask(bitmask);

info.setPcAtFault(pc);

info.category = ExceptionCategory::INVALID_OPERATION; 

argCpu->raiseException(info);

  return;

}

Summary: Traps are delivered only if both the status bit is set and the corresponding FPCR enable bit is set.

 

Trap Always

 

(OptimizedExceptionStatusRegister Only; Unconditional Trap)

Used for: /U, /UM, /UC, /UD

 

// Trap always if exception occurred, regardless of FPCR trap enables

if (status->hasInvalidOpOccurred()) {

    cpu->raiseException(ExceptionCategory::INVALID_OPERATION, pc, 0); return;

}

if (status->hasOverflowOccurred()) {

    cpu->raiseException(ExceptionCategory::FLOATING_POINT_OVERFLOW, pc, 0); return;

}

if (status->hasUnderflowOccurred()) {

    cpu->raiseException(ExceptionCategory::FLOATING_POINT_UNDERFLOW, pc, 0); return;

}

if (status->hasInexactOccurred()) {

    cpu->raiseException(ExceptionCategory::INEXACT, pc, 0); return;

}

 

Summary: Traps are delivered for any occurred exception—FPCR is not checked.

 

Status Only (Masked/Summary-Only)

 

(OptimizedExceptionStatusRegister Only; Never Traps)

Used for: /M, /SUM, /SUID, etc.

 

// Status only: update OptimizedExceptionStatusRegister, never trap

status->testInvalidOccurred(dtResult);

status->testOverflowOccurred(dtResult);

status->testUnderflowOccurred(dtResult);

status->testInexactOccurred(dtA, dtResult);

// No RaiseException called

 

Summary: Never triggers a trap/exception; only sets status bits. Do not Call AlphaCPU::raiseException()

 

About RaiseException:

 

•RaiseException is the single, canonical entry point for actually delivering an architectural trap/exception.

•The above logic ensures the grain decides whether to deliver a trap.

•It does so by checking:

oFor "Trap Only if Both": Exception status and FPCR trap enable

oFor "Trap Always": Exception status only (no FPCR check)

oFor "Status Only": Never calls RaiseException (status update only)

•Once the trap is to be delivered, RaiseException takes over: saves context, updates architectural state, possibly signals PALcode, etc.

Grain Implementation and Usage:

•Always update OptimizedExceptionStatusRegister status bits after the operation.

•Select the right trap pattern for the grain type (per your table).

•Only then write result, notify, and advance PC.

 

Summary

 

•All architectural "trap" logic for FP arithmetic is handled by the combination of OptimizedExceptionStatusRegister (status bits) and FPCR (trap enables) per instruction model.

•RaiseException is only called if the grain's model and current status/enable bits require a trap.
